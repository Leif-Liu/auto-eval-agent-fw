# Defect Description Agent 能力评估方案提案

**Evaluation Framework Proposal**

版本 v3.0 | 2026/5/7

---

## 1. 背景与目标

本提案针对 Defect Description Agent 的能力评估需求，制定系统性评估框架。该 Agent 的核心功能为：

1. 接收用户输入的 defect description 作为原始输入
2. 处理输入内容并输出结构化的 Summary 建议
3. 识别并解决 defect description 中存在的冲突（Conflict）
4. 检测并修正语法错误（Grammar Error）

鉴于 Agent 在构建完成后需要时间逐步"成长"与优化，建立一套客观、可量化、可持续追踪的评估体系尤为重要，其核心目标为：

1. 量化 Agent 各维度的当前能力水平，建立基线（Baseline）
2. 追踪 Agent 随迭代优化的能力成长曲线
3. 识别能力瓶颈，为优化方向提供数据依据
4. 为项目里程碑的判断提供客观标准

---

## 2. 评估维度与指标体系

根据 Agent 的三大核心功能，评估体系划分为五个维度，各维度权重根据业务重要性设定：

| 评估维度 | 核心指标（原始值） | 评估方法 | 百分制换算规则（0-100分） | 权重 |
| --- | --- | --- | --- | --- |
| Summary 准确性 | · 语义准确性（0~100）· 结构字段正确性（0~100）· 概括质量（0~100）· 信息完整性（0~100） | LLM-as-a-Judge（大模型评审）对比黄金测试集 | 四子维度由评审 LLM 直接按百分制打分，加权求和：得分 = 语义准确×0.35 + 字段正确×0.30 + 概括质量×0.20 + 信息完整×0.15。例：90×0.35+85×0.30+88×0.20+82×0.15 = 86.6分 | 30% |
| 冲突检测 | · 召回率 Recall（0~1）· 精确率 Precision（0~1） | 预设冲突测试集专家评审 | 用F1综合两者，避免单指标偏斜：F1 = 2×(P×R)/(P+R)，得分 = F1×100。例：P=0.88，R=0.92 → F1=0.90 → 90分 | 25% |
| 语法纠错 | · 修复率（0~1）· 误改率（0~1，越低越好） | 语法错误注入测试 | 修复率得分 = 修复率×100；误改率得分 = (1−误改率)×100；最终得分 = 修复率得分×0.7 + 误改率得分×0.3。例：修复率=0.90，误改率=0.05 → 90×0.7+95×0.3 = 91.5分 | 20% |
| **输出质量** | **· 专家盲评（0~100分）· 子维度：流畅性 / 专业性 / 格式规范** | **人工盲评（2-3名专家）** | **三子维度直接按百分制打分，加权求和：得分 = 流畅×0.3 + 专业×0.5 + 格式×0.2。例：流畅=85，专业=70，格式=90 → 85×0.3+70×0.5+90×0.2 = 78.5分** | **15%** |
| 系统稳定性 | · 异常输入处理率（0~1）· 推理效率达标率（0~1） | 边界用例测试推理日志分析 | 两项加权合并：得分 = (异常处理率×0.5 + 推理效率达标率×0.5)×100。例：0.95，0.90 → 92.5分 | 10% |
| **合计** | — | — | 权重之和 = 30+25+20+15+10 = 100%，各维度得分均为 0~100 分 | **100%** |

**【设计原则】** 所有维度原始指标统一换算为 0~100 分后再参与加权，确保公式逻辑自洽、结果可比。

### Summary 准确性 — LLM-as-a-Judge 评估说明

采用独立的大模型（LLM）作为评审，将 **黄金测试集中的标准 Summary（Ground Truth）** 与 **Agent 生成的 Summary** 进行语义层面的对比评分，替代传统的 ROUGE-L 等词面匹配指标。

**评审 Prompt 模板：**

```
你是一个 Defect Description 领域的评审专家。
请对比【Ground Truth Summary】和【Agent 生成的 Summary】，从以下维度打分（0-100）：

1. 语义准确性：Agent Summary 是否准确传达了 Ground Truth 的核心含义？
   是否存在语义偏差或误解？
2. 结构字段正确性：[Product Line][VCU][Security Level][Build Flavor][Vehicle Program]
   各字段与 Ground Truth 是否一致？
3. Defect Description 概括质量：缺陷描述部分是否简洁准确？
   与 Ground Truth 的表述是否语义等价？
4. 信息完整性：是否遗漏了 Ground Truth 中的关键信息？
   是否引入了 Ground Truth 中不存在的错误信息？

【Ground Truth Summary】：{ground_truth_summary}
【Agent 生成的 Summary】：{generated_summary}

请逐维度打分并给出简要理由。
```

**得分计算：**

$$\text{Summary准确性得分} = \text{语义准确} \times 0.35 + \text{字段正确} \times 0.30 + \text{概括质量} \times 0.20 + \text{信息完整} \times 0.15$$

**权重设计说明：**

| 子维度 | 权重 | 理由 |
| --- | --- | --- |
| 语义准确性 | 0.35 | 核心指标，Summary 的首要目的是准确传达语义 |
| 结构字段正确性 | 0.30 | 结构化字段（Product Line、VCU 等）的提取准确性直接影响后续流程 |
| 概括质量 | 0.20 | 缺陷描述的简洁性与准确性影响可读性和效率 |
| 信息完整性 | 0.15 | 确保无遗漏、无幻觉信息 |

**计算示例：**

| 子维度 | 评审 LLM 打分 | 权重 | 加权得分 |
| --- | --- | --- | --- |
| 语义准确性 | 90 | 0.35 | 31.5 |
| 结构字段正确性 | 85 | 0.30 | 25.5 |
| 概括质量 | 88 | 0.20 | 17.6 |
| 信息完整性 | 82 | 0.15 | 12.3 |
| **合计** |  | **1.0** | **86.9 分** |

**评估一致性保障措施：**

1. **温度设为 0** — 降低 LLM 输出的随机性
2. **多次评审取均值** — 同一样本运行 3 次，取平均分
3. **校准集验证** — 用 20-30 条已有人工评分的样本做校准，确保 LLM 评分与人工评分的相关性 ≥ 0.85
4. **定期人工抽检** — 每轮随机抽取 10% 样本由人工复核 LLM 评分

### 冲突检测指标定义

| 术语 | 预测结果 | 实际结果 | 含义及在评估公式中的作用 |
| --- | --- | --- | --- |
| TP 真阳性（True Positive） | 冲突 | 冲突 | 正确检测出的冲突。参与 Precision 分子 & Recall 分子 |
| FP 假阳性（False Positive） | 冲突 | 无冲突 | 误报的冲突（多报）。拉低 Precision：Precision = TP/(TP+FP) |
| FN 假阴性（False Negative） | 无冲突 | 冲突 | 漏掉的冲突（少报）。拉低 Recall：Recall = TP/(TP+FN) |
| TN 真阴性（True Negative） | 无冲突 | 无冲突 | 正确排除的非冲突。不参与 Precision/Recall，反映误报抑制能力 |

- **Precision** = TP / (TP + FP) — 在所有预测为冲突的样本中，确实是冲突的比例（衡量误报）
- **Recall** = TP / (TP + FN) — 在所有真实冲突中，被正确检测出的比例（衡量漏报）
- **F1** = 2 × (Precision × Recall) / (Precision + Recall) — Precision 与 Recall 的调和平均，作为冲突检测维度的最终得分

调和平均的两个关键特性：

1. **惩罚极端值：** 当 Precision 或 Recall 任意一项极低时，F1 会被大幅拉低，远低于两者的算术平均值。例：Precision=0.95，Recall=0.10 → 算术均值=52.5%，F1=18.1%
2. **要求平衡：** 只有当 Precision 与 Recall 同时较高时，F1 才会较高，确保 Agent 既不过度误报、也不大量漏报。例：Precision=0.88，Recall=0.92 → F1=90.0%

### 语法纠错指标定义

| 指标 | 原始数据计算方式 | 含义说明 |
| --- | --- | --- |
| 修复率（Correction Rate） | = 被正确修复的错误数 / 注入的总错误数 | 衡量 Agent 找到并修复错误的能力。例：注入20处错误，正确修复18处 → 修复率 = 18/20 = 0.90 |
| 误改率（Over-correction Rate） | = 被错误修改的正确词数 / 原文中正确词的总数 | 衡量 Agent 误将正确内容改错的比例。例：原文100个正确词，被误改5个 → 误改率 = 5/100 = 0.05 |

百分制换算与最终得分：

- 修复率得分 = 修复率 × 100（直接将原始修复率放大至百分制）
- 误改率得分 = (1 − 误改率) × 100（转为正确保留率方向，误改率越低得分越高）
- **语法纠错最终得分 = 修复率得分×0.7 + 误改率得分×0.3**
    - 例：修复率=0.90，误改率=0.05 → 修复率得分=90，误改率得分=95 → 最终得分 = 90×0.7 + 95×0.3 = 91.5分 ✓ 符合直觉

### 输出质量得分 — 计算说明

专家直接在 **0–100 分** 百分制上对每个子维度独立打分，然后加权求和：

$$\text{输出质量得分} = \text{流畅} \times 0.3 + \text{专业} \times 0.5 + \text{格式} \times 0.2$$

**权重设计说明：**

| 子维度 | 权重 | 理由 |
| --- | --- | --- |
| 流畅性 | 0.3 | 语句通顺、可读性好是基本要求 |
| 专业性 | 0.5 | Defect Description 场景下，术语准确、内容专业是核心质量指标 |
| 格式规范 | 0.2 | 格式统一有助于可读性，但相对于内容质量优先级较低 |

#### 计算示例

**示例 1：专业性较弱的情况**

| 子维度 | 专家打分 | 权重 | 加权得分 |
| --- | --- | --- | --- |
| 流畅性 | 85 | 0.3 | 25.5 |
| 专业性 | 70 | 0.5 | 35.0 |
| 格式规范 | 90 | 0.2 | 18.0 |
| **合计** |  | **1.0** | **78.5 分** |

> 尽管流畅性和格式都较高，但专业性仅 70 分（权重最大），总分被明显拉低到 78.5，如实反映了专业性短板。

**示例 2：均衡表现的情况**

| 子维度 | 专家打分 | 权重 | 加权得分 |
| --- | --- | --- | --- |
| 流畅性 | 82 | 0.3 | 24.6 |
| 专业性 | 88 | 0.5 | 44.0 |
| 格式规范 | 85 | 0.2 | 17.0 |
| **合计** |  | **1.0** | **85.6 分** |

> 三项表现均衡且专业性突出，总分 85.6，与直觉相符。

### 系统稳定性指标定义

$$\text{系统稳定性得分} = (\text{异常处理率} \times 0.5 + \text{推理效率达标率} \times 0.5) \times 100$$

#### 1. 异常输入处理率

衡量 Agent 在面对非标准、边界或恶意输入时，能否稳定运行而不崩溃、不输出无意义结果。

**计算公式：**

$$\text{异常处理率} = \frac{\text{正确处理的异常用例数}}{\text{总异常测试用例数}}$$

**"正确处理"的判定标准** — 满足以下任一条即算通过：

| 处理方式 | 说明 | 示例 |
| --- | --- | --- |
| 优雅降级 | 无法完成完整任务时，返回部分有效结果并附说明 | 输入关键信息均冲突不相关，Agent 无法抉择提示用户作出选择 |
| 明确错误提示 | 返回清晰、有意义的错误信息，指引用户修正输入 | 输入为空时，返回"请输入有效的 defect description" |

**"未正确处理"的判定标准** — 出现以下任一情况即算未通过：

| 失败类型 | 说明 |
| --- | --- |
| 返回无意义内容 | 输出乱码、空白或与输入完全无关的内容 |
| 静默吞错 | 未报错也未给出任何提示，用户无法感知异常 |

**异常测试用例类型：**

| 编号 | 用例类型 | 典型输入 |
| --- | --- | --- |
| E1 | 超长输入 | 超过 10000 字的 defect description |
| E2 | 纯乱码 | 随机字符串、特殊符号序列 |
| E3 | 格式异常 | 缺少必要字段、JSON 格式错误 |
| E4 | 跨语言混合 | 中英混杂输入 |


**计算示例：** 共设计 40 个异常测试用例，Agent 正确处理了 38 个 → 异常处理率 = 38/40 = **0.95**

#### 2. 推理效率达标率

衡量 Agent 在处理请求时能否有效复用已有推理信息，避免重复推理，提升响应效率。

**核心思想：** 当用户对同一条 defect description 进行增量修改或追问时，Agent 应能识别当前上下文状态，复用之前的推理结果（如已提取的字段、已检测的冲突等），仅对变化部分进行增量推理，而不是每次都从头开始完整推理。

**计算公式：**

$$\text{推理效率达标率} = \frac{\text{推理效率达标的请求数}}{\text{总测试请求数}}$$

**达标判定标准：**

| 场景 | 达标 ✅ | 不达标 ❌ |
| --- | --- | --- |
| 用户修改了 description 中的一个字段 | Agent 识别变化部分，仅更新受影响的字段和 Summary，复用其余推理结果 | Agent 忽略之前推理，对所有字段重新完整推理 |
| 用户追问"冲突检测结果是否变化" | Agent 基于已有状态直接回答，无需重新执行全量冲突检测 | Agent 从头重新运行完整的冲突检测流程 |
| 用户未修改输入，要求生成简化 Summary | Agent 复用已有推理信息，快速返回结果 | Agent 完全丢弃之前的推理上下文，全量重算 |
| 输入内容完全变更（新的 defect description） | Agent 正确判断需要全新推理，执行完整流程 | 无法识别全新的 defect description |

**评估方法：** 通过分析 Agent 的推理日志（Reasoning Log），判断每次请求是否存在不必要的重复推理步骤：

1. **设计增量修改测试序列** — 对同一条 defect description 进行多轮渐进修改（如先改字段、再改描述、再追问），记录每轮 Agent 的推理步骤
2. **对比推理步骤** — 检查 Agent 是否在后续轮次中重复执行了前序轮次已完成且未受影响的推理步骤
3. **判定达标** — 若 Agent 能正确识别变化范围并仅做增量更新，即为达标

**计算示例：** 设计 20 条增量修改测试序列，每条包含 3-5 轮追问，共产生 80 个增量请求。其中 72 个请求 Agent 正确复用了已有推理信息 → 推理效率达标率 = 72/80 = **0.90**

#### 综合得分计算示例

| 指标 | 值 | 权重 | 加权 |
| --- | --- | --- | --- |
| 异常处理率 | 0.95 | 0.5 | 0.475 |
| 推理效率达标率 | 0.90 | 0.5 | 0.450 |
| **合计** |  |  | **0.925 × 100 = 92.5 分** |

### 综合能力得分（满分100分）计算公式

$$\text{总分} = \text{Summary准确性} \times 0.30 + \text{冲突检测} \times 0.25 + \text{语法纠错} \times 0.20 + \text{输出质量} \times 0.15 + \text{系统稳定性} \times 0.10$$

**【计算示例】** 某次评估结果：Summary准确性=86分，冲突检测=90分，语法纠错=75分，输出质量=78.5分，系统稳定性=97分

> 总分 = 86×0.30 + 90×0.25 + 75×0.20 + 78.5×0.15 + 97×0.10 = 25.8 + 22.5 + 15.0 + 11.775 + 9.7 = **84.8分** → L3 成熟级

## 3. 评估方法详解

### 3.1 黄金测试集（Golden Test Set）

构建覆盖核心场景的固定测试数据集，作为每次迭代的标准化基准：

1. **数据规模：** 建议初始集≥200条，涵盖不同类型、长度、复杂度的 defect description
2. **标注内容：** 每条数据包含标准 Summary、已知冲突标注、语法错误标注及期望输出
3. **分层设计：** 按难易度划分（简单 / 中等 / 复杂），确保覆盖边界情况。其中"复杂"层应包含以下极端场景：
    - 同时含多个相互矛盾冲突的 defect description — 测试复杂冲突解析能力
    - 严重语法错误导致语义模糊的输入 — 测试极端条件下的语义理解与纠错能力
4. **更新机制：** 每季度评审并补充新场景，保持测试集的代表性

### 3.2 自动化评估（Automated Evaluation）

利用自动化工具实现快速、低成本的迭代评估：

1. **Summary 质量：** 采用 LLM-as-a-Judge 方法，由评审 LLM 对比 Ground Truth Summary 与 Agent 生成的 Summary，从语义层面进行多维度评分（指标定义详见第2节）
2. **冲突检测：** 基于预设冲突用例，通过 TP / FP / FN / TN 四类结果计算 Precision、Recall 及 F1 分数（指标定义详见第2节）
3. **语法纠错：** 注入已知语法错误，分别统计修复率与误改率，各自独立换算为百分制后加权合并（指标定义详见第2节）
4. **系统稳定性：** 边界用例测试（异常处理率）与推理日志分析（推理效率达标率），指标定义详见第2节

### 3.3 人工评估（Human Evaluation）

针对输出质量等主观维度，引入领域专家进行盲评：

1. **评审人员：** 建议由 2-3 名熟悉 defect description 领域的专家参与
2. **评分方式：** 专家对每个子维度直接在 **0–100 分** 百分制上独立打分（参照锚定表），按加权公式计算输出质量得分：得分 = 流畅×0.3 + 专业×0.5 + 格式×0.2
3. **盲评机制：** 评审人员不知晓输出来源，确保评分客观性
4. **评审频次：** 每月进行一次，每次抽取50-100条输出样本

## 4. 评估流程与实施方案

本节基于 Agent System Development Life Cycle，聚焦 **Testing & Evaluation（阶段04）→ Monitoring & Ops（阶段07）→ Iteration（迭代优化）** 的完整评估闭环，细化各步骤的具体操作、输入输出和责任方。

### 4.1 评估流程总览


### 4.2 STEP 1：构建验证数据集（第1-2周）

**对应生命周期阶段：** 04 Testing & Evaluation

**目标：** 建立标准化的黄金测试集（Golden Test Set），作为所有后续评估的固定基准。

**具体操作：**

| 步骤 | 操作内容 | 负责方 | 输出 |
| --- | --- | --- | --- |
| 1.1 数据收集 | 从历史 defect description 中筛选 ≥200 条典型样本，覆盖不同类型、长度、复杂度 | 测试团队 + 业务专家 | 原始数据集 |
| 1.2 分层标注 | 按难易度分层（简单/中等/复杂），复杂层包含多冲突、严重语法错误等极端场景 | 业务专家（2-3人） | 分层标注的数据集 |
| 1.3 Ground Truth 标注 | 为每条数据标注标准 Summary、冲突标注、语法错误标注及期望输出 | 业务专家交叉标注 | 黄金测试集 v1.0 |
| 1.4 异常用例构建 | 设计 ≥40 条异常测试用例（E1-E4：超长输入、纯乱码、格式异常、跨语言混合） | 测试团队 | 异常测试用例集 |
| 1.5 增量测试序列 | 设计 ≥20 条增量修改测试序列（每条 3-5 轮追问），用于推理效率达标率评估 | 测试团队 | 推理效率测试集 |

**质量标准：**
- 黄金测试集需至少 2 名专家独立标注，标注一致率 ≥ 85%
- 数据分布应覆盖所有 Product Line、VCU 类型

### 4.3 STEP 2：建立基线（第2-3周）

**对应生命周期阶段：** 04 Testing & Evaluation → Quality Gate Passed

**目标：** 用验证数据集对当前 Agent 进行首次全量评估，得到五个维度的初始基线得分，并设定各维度目标得分。

**具体操作：**

| 步骤 | 操作内容 | 输出 |
| --- | --- | --- |
| 2.1 自动化评估运行 | 将黄金测试集全量输入 Agent，收集生成结果；运行 LLM-as-a-Judge、F1 计算、修复率/误改率统计 | 五维度原始得分 |
| 2.2 人工盲评 | 2-3 名专家对 Agent 输出进行流畅性/专业性/格式规范的百分制打分 | 输出质量得分 |
| 2.3 系统稳定性测试 | 运行异常用例集 + 增量测试序列，统计异常处理率和推理效率达标率 | 系统稳定性得分 |
| 2.4 综合得分计算 | 按加权公式计算总分，确定当前 L1-L4 成熟度等级 | 基线评估报告 |
| 2.5 设定目标得分 | 结合业务需求，为每个维度设定达标目标分（如 Summary 准确性 ≥ 85） | 目标得分基准表 |

**输出示例 — 基线 vs 目标对照表：**

| 评估维度 | 基线得分（现状） | 目标得分（期望） | 差距 | 优先级 |
| --- | --- | --- | --- | --- |
| Summary 准确性 | 72 | ≥ 85 | -13 | 🔴 高 |
| 冲突检测 | 65 | ≥ 80 | -15 | 🔴 高 |
| 语法纠错 | 58 | ≥ 75 | -17 | 🔴 高 |
| 输出质量 | 70 | ≥ 80 | -10 | 🟡 中 |
| 系统稳定性 | 80 | ≥ 90 | -10 | 🟡 中 |

**Quality Gate 判定：** 综合得分 ≥ 60 分（L2 及以上）方可进入 Integration & Staging 阶段。

### 4.4 STEP 3：Go-live 部署上线

**对应生命周期阶段：** 05 Integration & Staging → 06 Deployment → Go-live

通过 Quality Gate 后，Agent 进入集成测试和部署上线流程。此步骤不展开（属于 DevOps 范畴），关键里程碑为：

- **UAT Signed-off** — 用户验收测试通过
- **Go-live** — Agent 正式上线生产环境
- **Monitoring Baseline** — 上线后建立生产环境监控基线

### 4.5 STEP 4：监控与运维（Monitoring & Ops）— 持续进行

**对应生命周期阶段：** 07 Monitoring & Ops

**目标：** 上线后持续监控 Agent 的运行质量，及时发现质量漂移（Quality Drift）。

#### 4.5.1 可观测性（Observability）建设

| 监控维度 | 监控指标 | 采集方式 | 告警阈值 |
| --- | --- | --- | --- |
| 请求量与可用性 | 日请求量、成功率、超时率 | Agent 运行日志 | 成功率 < 90% 告警 |
| 推理效率 | 平均推理步骤数、重复推理比例 | 推理日志（Reasoning Log） | 重复推理比例 > 20% 告警 |
| 输出质量抽检 | 每日随机抽样 10-20 条输出，由 LLM-as-a-Judge 自动评分 | 自动化评审管线 | 平均分连续 3 天低于基线 5 分以上告警 |
| 用户反馈 | 用户对 Agent 输出的采纳率、修改率、拒绝率 | 前端埋点 | 采纳率 < 60% 告警 |
| 异常处理 | 异常输入占比、异常处理成功率 | Agent 运行日志 | 异常处理成功率 < 90% 告警 |

#### 4.5.2 质量漂移检测（Quality Drift Detection）

质量漂移指 Agent 上线后性能逐渐下降的现象，常见原因包括：

| 漂移类型 | 表现 | 检测方式 |
| --- | --- | --- |
| 数据漂移 | 新输入的 defect description 风格/术语与训练数据差异增大 | 对比输入分布与黄金测试集分布 |
| 模型漂移 | LLM 底座更新后输出风格或质量变化 | 每次模型更新后重跑黄金测试集对比得分 |
| Prompt 退化 | 多次修改 prompt 后累积副作用 | 版本化管理 prompt，每次变更后回归测试 |

**检测节奏：**
- **每日：** 自动抽样评分，监控趋势
- **每周：** 汇总周报，分析趋势变化
- **触发式：** 任何模型/Prompt 变更后，立即重跑黄金测试集

### 4.6 STEP 5：收集原始测试数据 & 导入需求（持续进行）

**对应生命周期阶段：** 07 Monitoring & Ops → Iteration 过渡

**目标：** 从生产环境中系统化收集 Agent 的问题案例和改进需求，形成优化输入。

#### 4.6.1 原始测试数据收集

| 数据来源 | 收集内容 | 收集方式 | 频率 |
| --- | --- | --- | --- |
| 自动化监控 | LLM-as-a-Judge 评分低于 60 分的 case | 自动标记 + 归档 | 实时 |
| 用户反馈 | 用户修改/拒绝的 Agent 输出（含修改前后对比） | 前端埋点 + 日志 | 实时 |
| 质量漂移告警 | 触发告警的具体 case 和上下文 | 告警系统自动归档 | 触发时 |
| 人工抽检 | 专家月度盲评中发现的典型问题案例 | 人工标注 + 归档 | 每月 |

#### 4.6.2 需求导入与问题归档

收集到的原始数据需要结构化归档，形成可执行的优化需求：

| 归档字段 | 说明 | 示例 |
| --- | --- | --- |
| Case ID | 唯一标识 | EVAL-2026-0507-001 |
| 影响维度 | 对应五个评估维度中的哪个 | Summary 准确性 |
| 问题描述 | 具体什么地方出了问题 | 结构字段 [Security Level] 提取错误 |
| 原始输入 | 触发问题的 defect description | （原文） |
| Agent 输出 | Agent 生成的错误结果 | （原文） |
| 期望输出 | 正确的 Ground Truth | （标注） |
| 优先级 | 影响面 × 频率 | P1 / P2 / P3 |
| 根因分析 | Prompt 问题 / 模型能力 / 数据分布 | Prompt 未覆盖此类场景 |

### 4.7 STEP 6：优化 Agent，迭代开发（每 2 周一个迭代）

**对应生命周期阶段：** Iteration

**目标：** 基于收集到的问题和需求，有针对性地优化 Agent，并通过回归测试验证优化效果。

#### 4.7.1 优化方向对照表

| 优化手段 | 对应生命周期要素 | 适用场景 | 验证方式 |
| --- | --- | --- | --- |
| **Prompt Optimization** | Prompt 优化 | 特定场景 Summary 不准确、字段提取错误、格式不规范 | 重跑相关子集测试用例，对比优化前后得分 |
| **Agent Capability Expansion** | 能力扩展 | 新增 Product Line 支持、新冲突类型识别、多语言适配 | 扩充黄金测试集，新增对应测试用例 |
| **Data Flywheel** | 数据飞轮 | 生产中发现的新型 case 反哺测试集，持续提升覆盖度 | 详见 STEP 7 |

#### 4.7.2 迭代开发流程

```
问题归档（STEP 5）
    │
    ▼
优先级排序 → 选取本迭代修复的 Top-N 问题
    │
    ▼
开发修复（Prompt 调整 / 工具调用优化 / 流程改进）
    │
    ▼
回归测试 → 重跑黄金测试集全量 + 新增问题用例
    │
    ▼
对比得分 → 与上一轮基线对比五维度得分变化
    │
    ├─ 得分提升且无回归 → 合并发布，更新基线
    │
    └─ 出现回归 → 回滚修改，分析原因，重新调整
```

#### 4.7.3 迭代评估报告（每 2 周输出）

| 报告内容 | 说明 |
| --- | --- |
| 五维度得分对比 | 本轮 vs 上轮 vs 基线 vs 目标，可视化趋势图 |
| 本轮修复问题清单 | 修复的 Case ID、影响维度、修复前后得分变化 |
| 新发现问题 | 本轮测试或监控中新发现的问题 |
| 下轮优化建议 | 基于得分差距，建议下一迭代的优化重点 |
| 成熟度等级变化 | L1→L2→L3→L4 的进展追踪 |

### 4.8 STEP 7：数据飞轮闭环更新（持续进行）

**对应生命周期阶段：** Iteration — Data Flywheel

**目标：** 将生产环境中收集的真实 case 持续反哺到验证数据集中，形成"评估 → 发现问题 → 优化 → 扩充测试集 → 重新评估"的正向飞轮。

**数据飞轮运转机制：**

```
生产环境真实 case
    │
    ▼
筛选有价值 case（低分 case、用户修改 case、新场景 case）
    │
    ▼
专家标注 Ground Truth → 纳入黄金测试集
    │
    ▼
黄金测试集版本升级（v1.0 → v1.1 → v2.0 ...）
    │
    ▼
重跑全量评估 → 更新基线
    │
    └──────> 回到 STEP 2，新基线成为下一轮迭代的对标参考
```

**测试集更新规则：**

| 规则 | 说明 |
| --- | --- |
| 新增频率 | 每迭代周期（2周）新增 10-20 条来自生产环境的真实 case |
| 季度大更新 | 每季度全面评审测试集，移除过时用例，补充新场景 |
| 版本管理 | 测试集严格版本化，每次变更记录变更日志，确保评估结果可追溯 |
| 分布平衡 | 新增用例后检查难易度分布（简单/中等/复杂 ≈ 3:4:3），避免偏斜 |

### 4.9 评估节奏总表

| 阶段 | 时间节点 | 对应生命周期 | 评估重点 | 预期输出 |
| --- | --- | --- | --- | --- |
| 构建验证数据集 | 第1-2周 | 04 Testing & Evaluation | 建立黄金测试集、异常用例集、增量测试序列 | 验证数据集 v1.0 |
| 建立基线 | 第2-3周 | 04 → Quality Gate | 首次全量评估，确认各维度初始得分（基线）；根据业务需求设定各维度目标得分 | 基线评估报告；能力雷达图(初版)；目标得分基准表 |
| 部署上线 | 第4周 | 05-06 → Go-live | 集成测试、UAT、生产部署 | UAT 报告；Go-live 确认 |
| 监控与运维 | 上线后持续 | 07 Monitoring & Ops | 可观测性监控；质量漂移检测；自动化抽样评分 | 日报/周报；告警记录 |

| 迭代评估 | 每2周一次 | Iteration | Prompt 优化 / 能力扩展 → 回归测试 → 对比得分 | 迭代对比报告；成长趋势图 |

### 4.10 能力成熟度评级

基于综合评分，定义 Agent 的能力成熟度等级，为项目决策提供参考：

| 等级 | 综合得分 | 状态描述 |
| --- | --- | --- |
| L1 - 初级 | < 60分 | 基础功能可用，存在明显缺陷，需持续优化 |
| L2 - 成长 | 60-74分 | 核心功能稳定，边界场景处理有待提升 |
| L3 - 成熟 | 75-89分 | 各维度表现良好，可用于生产环境 |
| L4 - 优秀 | ≥ 90分 | 全维度高水平，具备行业竞争力 |
